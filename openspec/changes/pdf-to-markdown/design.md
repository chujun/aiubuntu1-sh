## Context

微信读书 iPad 横屏双页模式截图生成的 PDF，每页为一张 1536x948 的 JPEG 图片，深色背景白色文字，无文字层。通过分析《独立宣言》（11页，纯文字）和《分布式系统架构》（183页，含架构图/表格）两本样本书，确认了截图格式的稳定性：
- UI 元素位置固定：顶部书名栏、底部"< 上一页 / 下一页 >"翻页提示
- 双页布局：左右页中间有明显间距
- 图片类型：架构图、流程图、统计图、代码截图，带"图X-Y 标题"格式图注
- 文字渲染清晰度高，优于传统扫描书籍

## Goals / Non-Goals

**Goals:**

- 文字一字不改还原为 Markdown，准确率目标 99.9%+
- 书中插图/图表裁剪保存为独立图片文件，Markdown 以相对路径引用
- 零 API 成本，完全本地运行
- 双引擎交叉校验，自动标记差异位置供人工审核
- 支持单本和批量处理
- 可复用于 10+ 本书的持续使用场景

**Non-Goals:**

- 不识别图片内部的文字（如架构图中的标注文字）
- 不做 PDF 原生文字层提取（输入 PDF 全是图片）
- 不支持非微信读书格式的 PDF（可后续扩展）
- 不做云端部署或 Web 界面
- 不做自动章节拆分为多文件（统一输出单个 book.md）

## Decisions

### D1: PDF 图片提取方式 — PyMuPDF 直接提取嵌入图片

**选择**: `fitz.Document.extract_image()` 直接提取 PDF 中嵌入的 JPEG

**替代方案**:
- `pdf2image` (poppler): 将 PDF 页面渲染为图片。需要安装 poppler 系统依赖，且渲染过程有质量损失。
- `PyMuPDF page.get_pixmap()`: 渲染页面为位图。同样有渲染损失。

**理由**: 微信读书 PDF 每页恰好嵌入一张 JPEG，直接提取是无损的，且不需要额外系统依赖。经样本验证，两本书均为此格式。

### D2: 图像预处理管道 — 规则裁剪 + 垂直投影分割

**UI 裁剪策略**: 基于固定像素比例裁剪顶部和底部 UI 区域。微信读书 UI 布局在同一设备上高度一致，通过配置参数适配不同设备。

**双页分割策略**: 对裁剪后图片做垂直投影（逐列统计像素亮度），在图片水平中间区域寻找亮度谷值（深色间隔），确定中线位置进行分割。

**颜色反转策略**: 将深色背景白色文字反转为白底黑字。仅对文字区域做反转，图片区域保持原始颜色（由版面分析后决定）。

**替代方案**:
- 基于深度学习的 UI 检测：过重，规则方式已足够。
- 固定中点分割：不够精确，左右页宽度可能不完全对称。

### D3: 版面分析 — PaddleOCR PP-Structure

**选择**: PP-Structure（PP-Layout + PP-OCR）

**理由**:
- 能区分文字、标题、图片、表格、公式 5 种区域类型
- 中文场景训练充分，与 PP-OCR 无缝集成
- Apache 2.0 开源协议，无商用限制

**替代方案**:
- LayoutParser：社区活跃度下降，模型更新较慢
- DocTR：中文支持弱于 PaddleOCR
- 纯规则检测：无法可靠区分图片和密集文字

### D4: OCR 引擎 — PaddleOCR PP-OCRv4 (主) + Surya (副) 交叉校验

**选择**: 双引擎并行识别，逐行 diff 比对

**交叉校验逻辑**:
1. 两引擎分别对同一单页图片做 OCR
2. 按行对齐（基于 y 坐标重叠度匹配行）
3. 逐行文本比对：
   - 完全一致 → 直接采信
   - 不一致 → 默认采用 PaddleOCR 结果（中文准确率更高），同时插入 `<!-- REVIEW: paddle="X" surya="Y" -->` 注释
4. 统计差异率，输出审核摘要

**替代方案**:
- 单引擎 PaddleOCR：准确率 ~97%，无交叉校验，错字不易发现
- 大模型 API：准确率最高但违反零成本约束
- Tesseract：中文识别率不够高（~85%），排除

### ADR-001: OCR 引擎选型对比

**状态**: 已决定
**日期**: 2026-04-22
**决策者**: 项目所有者

#### 背景

项目核心需求是对微信读书截图（深色背景、白色中文文字、1536x948 JPEG）做 OCR 识别，要求：
- 中文识别准确率尽可能接近 100%（目标 99.9%+）
- 零 API 成本，完全本地运行
- 能处理中英文混排内容
- 支持与其他引擎交叉校验

#### 逐方案分析

**Tesseract 5 — ❌ 排除**

Tesseract 是最老牌的开源 OCR 引擎，但其中文识别率（~85%）远不能满足"一字不改"的要求。每 1000 字约 150 个错字，一本 6000 字的书会有约 900 个错字，人工校对成本极高。此外无版面分析能力，无法区分图片和文字区域。

**PaddleOCR PP-OCRv4 — ✅ 选为主引擎**

百度开发的深度学习 OCR 系统，中文识别率在开源方案中最高（~97%）。PP-OCRv4 使用了大规模中文数据训练，对中文排版的理解优于其他开源方案。配套的 PP-Structure 提供完整的版面分析能力。Apache 2.0 协议无商用顾虑。

优势：
- 中文识别率最高（开源方案中）
- PP-Structure 版面分析与 OCR 无缝集成
- 百度大规模中文训练数据
- 社区活跃，文档丰富

劣势：
- 安装 PaddlePaddle 依赖较重
- 阅读顺序检测需要额外处理

**Surya — ✅ 选为副引擎（交叉校验）**

由 VikParuchuri 开发的现代文档 OCR 系统，内置版面分析和阅读顺序检测。支持 90+ 语言。其架构与 PaddleOCR 完全不同（基于 Transformer），这使其作为交叉校验引擎特别有价值——两个不同架构的模型同时犯同一个错误的概率极低。

优势：
- 与 PaddleOCR 架构差异大，交叉校验互补性强
- 内置阅读顺序检测（双页场景有用）
- 安装简单（pip install surya-ocr）

劣势：
- 中文识别率略低于 PaddleOCR（~93% vs ~97%）
- GPL 3.0 协议（个人使用无影响，商用需注意）
- 模型较大（~500MB）

**EasyOCR — ⚠️ 备选方案**

如果未来需要规避 Surya 的 GPL 协议，EasyOCR（Apache 2.0）可作为替代的第二引擎。但其中文识别率（~90%）低于 Surya，交叉校验的价值会降低。

**多模态大模型 (Claude Vision / GPT-4o) — ❌ 排除（零成本约束）**

识别率最高（~98-99%），天然理解版面结构，能直接输出 Markdown。但违反零 API 成本的硬约束。保留为未来"高精度模式"的可选升级路径。

#### 决策

**主引擎**: PaddleOCR PP-OCRv4 — 中文识别率最高的开源方案，配套 PP-Structure 提供版面分析
**副引擎**: Surya — 与 PaddleOCR 架构互补，交叉校验时互补价值最大

**交叉校验策略**:
- 两引擎独立识别同一页面
- 逐行 diff 比对，一致则采信，不一致则标记 `<!-- REVIEW -->`
- 不一致时默认采用 PaddleOCR 结果（中文准确率更高）
- 预期效果：单引擎 ~97% → 双引擎交叉校验后可标记出大部分错误位置，人工只需审核标记位置

**预期校验效果估算**:
```
假设 PaddleOCR 准确率 97%，Surya 准确率 93%

每 1000 字:
  PaddleOCR 错 ~30 字, Surya 错 ~70 字

  两者同时出错且错成相同结果的概率极低（<0.1%）
  → 绝大多数错误会被交叉校验捕获
  → 人工审核量 = 两引擎不一致的行数（预计 5-8%）
  → 漏网错误 < 1字/1000字（99.9%+ 准确率目标可达）
```

#### 后果

- 需要同时安装 PaddlePaddle 和 Surya 两套依赖，磁盘占用约 700MB+
- 处理速度约为单引擎的 1.5 倍（可并行处理部分抵消）
- 项目因 Surya GPL 依赖，整体不适合闭源商业分发（个人使用无影响）
- 若未来需去除 GPL 依赖，可用 EasyOCR 替换 Surya，代码改动集中在 `ocr_engine.py`

### D5: Markdown 组装策略

**标题识别**: PP-Structure 标题区域 + 字体大小推断层级。微信读书的章节标题字号明显大于正文，可通过 OCR 文本框高度比例判断 `#` / `##` / `###`。

**段落拼接**:
- 同一页内：按 y 坐标顺序排列文字区域
- 跨页：检测左页末尾是否以句号/问号/感叹号结尾，若非则与右页开头拼接为同一段落

**图片引用**:
- 图片区域裁剪保存为 `images/page{页码}_fig{序号}.png`
- 若检测到"图X-Y"格式图注，用作 alt text：`![图3-3 架构图](images/page041_fig01.png)`

**表格处理**: PP-Structure 表格区域 → PaddleOCR 表格识别模块 → Markdown 表格语法

## Risks / Trade-offs

**[双页分割精度]** → 垂直投影法依赖中间间距的亮度差异。若某些截图左右页内容紧密无明显间隔，分割可能不准。→ **缓解**: 提供手动中线偏移配置参数；首次运行时输出分割预览供用户确认。

**[颜色反转对图片的影响]** → 全页反转会导致深色背景的图片变为不自然的亮色。→ **缓解**: 先做版面分析识别图片区域，仅对文字区域反转；图片区域保持原色裁剪保存。但这要求版面分析在反转前运行（深色图片上做版面分析）。

**[PP-Structure 版面分析精度]** → 深色背景可能影响 PP-Structure 的区域检测精度（训练数据以白底为主）。→ **缓解**: 对整页先做反转再做版面分析，然后用反转前的坐标回映射裁剪图片区域。

**[OCR 行对齐困难]** → PaddleOCR 和 Surya 的文本行切分粒度可能不同，导致逐行 diff 对不上。→ **缓解**: 按 y 坐标重叠度做模糊匹配，允许一定容差；对整页文本做全文 diff 作为兜底。

**[Surya GPL 协议]** → Surya 使用 GPL 3.0，若项目未来开源或商用需注意协议传染。→ **缓解**: 当前为个人工具，无影响。若需规避，可替换为 EasyOCR（Apache 2.0）作为第二引擎。

**[跨页段落拼接]** → 仅通过末尾标点判断是否跨页不够可靠（如引用、列表等场景）。→ **缓解**: 保守策略——默认不拼接，仅在明确检测到断句时拼接；用户可在后处理中手动调整。

## Open Questions

- **版面分析的执行顺序**: 是先反转再分析（白底更准），还是先分析再反转（避免图片被反转）？需要用样本实测对比。
- **特殊页面处理**: 版权页、目录页、空白页是否需要特殊逻辑跳过或标记？
- **配置化程度**: UI 裁剪的像素参数是否需要支持多设备配置文件（iPad / 手机 / PC）？当前仅针对 iPad 1536x948。
